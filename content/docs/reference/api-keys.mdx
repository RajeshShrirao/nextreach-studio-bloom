Pulseboard supports three key types: browser keys, server keys, and webhook delivery keys. Each one is scoped to a workspace and tracked in the audit log.

For demo purposes, the content is condensed into one page. In a real docs site this would likely be split into overview, permissions matrix, and troubleshooting.

## Key types

### Browser keys

Use for client-side tracking. These keys:
- Can only send events, not read data
- Are safe to expose in public code
- Start with `pk_` prefix

### Server keys

Use for server-side API calls. These keys:
- Can read and write workspace data
- Should be kept secret
- Start with `sk_` prefix

### Webhook keys

Use for verifying incoming webhooks. These keys:
- Are used to sign webhook payloads
- Should be kept secure
- Start with `wh_` prefix

## Rotation policy

<Steps>

1. Create separate keys for staging and production
2. Name each key after the service that uses it
3. Rotate any key that is exposed in logs or screenshots
4. Review audit entries after each rotation

</Steps>

<Callout type="warning" title="Security">
  Never commit API keys to version control. Use environment variables or secret management tools.
</Callout>

## Audit logs

All key operations are logged:
- Key creation
- Key rotation
- Key revocation
- Failed authentication attempts

Review audit logs regularly to detect unauthorized access.
